Technical summary

lfs-rl, a verifiable RL environment for systems-debugging agents

An agent diagnoses and repairs a fault injected into a fully-built Linux system. The reward is a functional check run inside the repaired system, not a string match.

RLVR·reward-hack-resistant·verifiers-compatible·50 fault instances·deterministic
Abstract

lfs-rl is a reinforcement-learning environment in which an agent diagnoses and repairs a fault injected into a fully-built Linux From Scratch system, using a shell and no hints. An out-of-band verifier runs a program inside the repaired system, so the reward resists gaming. With the included pipeline (supervised warm-start, then GRPO) Qwen2.5-3B goes from 0% to a 90% full-solve rate on its 13 trained fault types for roughly $0.50 of compute. Holdouts added after training measure 0% transfer on both axes, cleanly separating procedural mastery from genuine diagnosis. A frontier model solves the full set zero-shot.

1Environment

Each episode begins from a working Linux From Scratch (LFS) chroot with a single injected fault. The set contains 50 deterministic instances across five library packages and six fault classes, from toolchain sabotage and cross-package breaks to runtime-correctness faults where the library builds but returns wrong data. The agent gets a shell with sudo and no network, one command per turn. Episodes run in a rootless sandbox at about 1.6 seconds each, which makes on-policy RL over real package builds practical.

Example episode. A cross-package fault (symptom in one package, cause in another)
$ chroot /mnt/lfs make -C /sources/kmod-34.2
/usr/bin/ld: undefined reference to `gzread' # symptom: kmod won't link
$ ls -l /usr/lib/libz.so
libz.so -> libbz2.so.1.0 # cause: zlib's dev symlink is mispointed
$ ln -sf libz.so.1.3.1 /usr/lib/libz.so && make && make install
verifier: libkmod compiles, links, and runs (reward 1.0)

2Verifiable reward

A functional verifier scores each episode out-of-band. It compiles a small canary program inside the repaired system, runs it, and compares the output against ground truth, for example a compress/decompress round-trip. The reward reflects whether the repair actually works, not whether the expected files are present. A stub library that only fakes the version string does not pass. We implemented that attack ourselves and every verifier rejects it. A dense shaped reward over milestones gives a weak base policy gradient, and a binary reward is used for evaluation. Adding a package needs only a new canary, and an automated check proves each new fault both solvable (the reference fix passes) and biting (a naive rebuild does not) before it enters the set.

3Training result

We apply a supervised warm-start on 111 verifier-passed reference trajectories, then run 9 rounds of GRPO (mean shaped reward 0.31 to 0.88) on a single GPU at $0.16/hour. Instances are deterministic and the warm-start demonstrates each fault type, so Table 1 reports training-distribution mastery. The warm-start teaches the repairs but executes them unreliably. GRPO consolidates them into 90% reliable, verifier-passing repairs. Before training, the base model could not enter the chroot at all.

Table 1. Binary full-solve rate across the 13 trained fault types (the repaired system must run correctly, not merely build), Qwen2.5-3B.

The shaped rewards diagnose the failure modes. On the unseen package (shaped 0.06) the policy cannot even execute its procedure. On the trained package with an unseen fault (shaped 0.75) the procedure executes flawlessly, but the fault survives the rebuild and the policy never checks its own repair. This recipe produced a reliable executor of demonstrated repair procedures, not a debugger. Closing that gap is the open training question the generator's cheap package scaling exists to serve.

4Difficulty calibration

A frontier model (Claude) solved every fault we could construct in 8 to 11 commands, including silent wrong output with no build error, a fault a rebuild does not clear, and a defect planted in the source tarball. The current set is within frontier reach, so the environment's value today is in training and evaluating open models, where it resolves every rung of the capability gradient from 0% to 100%.

5Integration and cost

  • FormatPackaged as a verifiers environment and validated end-to-end against a live model endpoint.
  • DeterminismThe same seed reproduces the same fault and verifier.
  • FootprintRootless sandbox, ~1.6 s/episode, ~200 MB per instance. A full training run cost about $0.50 of GPU.
  • IncludedThe environment, fault generator, training pipeline, trained adapters, and calibration harness.
  • ScalingAdding zstd took a 56-line verifier and one taxonomy entry, and yielded nine new faults, each proven solvable and biting by CI. Eight further packages are staged.

6Fit and availability

lfs-rl is intended for teams and platforms that train or evaluate open models on agentic, coding, or operations tasks, and for RL-environment marketplaces that require verifiable, hard-to-game rewards.

The full environment, pipeline, adapters, and calibration data are available for evaluation. We welcome a technical review.
