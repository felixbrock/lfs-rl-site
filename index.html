Technical summary

lfs-rl, a verifiable RL environment for systems-debugging agents

An agent diagnoses and repairs a fault injected into a fully-built Linux system. The reward is a functional check run inside the repaired system, not a string match.

RLVR·reward-hack-resistant·verifiers-compatible·50 fault instances·deterministic
Abstract

We describe lfs-rl, a reinforcement-learning environment in which an agent must diagnose and repair a fault injected into a fully-built Linux From Scratch system, using a shell and no hints. The reward is verifiable and designed to resist reward hacking. Instead of comparing output text, an out-of-band verifier compiles and runs a program inside the repaired system and checks its behaviour. Using only the environment's own pipeline, a supervised warm-start followed by GRPO, we take Qwen2.5-3B from 0% to a 90% binary full-solve rate across its 13 training fault types, each solve a genuine, functionally-verified repair, for roughly $0.50 of compute. Holdouts added after training, an unseen package and unseen fault classes on a trained package, then measure 0% transfer on both axes. The environment cleanly separates procedural mastery from genuine diagnosis, and we report the full matrix. A frontier model solves the full fault set zero-shot in 8 to 11 commands, giving the capability gradient a measured span. It ships in the verifiers format, is deterministic, and runs without root or a VM.

1Environment

Each episode begins from a working Linux From Scratch (LFS) chroot and injects a single fault. The current set contains 50 deterministic instances across five library packages and one cross-package target, spanning six fault classes. These are toolchain sabotage, silent filesystem failures, missing development files, cross-package breaks (where the package that fails is not the one that is broken), runtime-correctness faults (the library builds and runs but returns wrong data), and composites of the above. sample_fault(seed) maps a seed to a fixed instance, so runs are reproducible.

The agent receives a shell with passwordless sudo and no network access, and acts one command per turn. Episodes run in a rootless sandbox (user, mount, and PID namespaces over an overlay copy-on-write rootfs) at about 1.6 seconds per episode, with no root privileges and no virtual machine. The throughput is what makes on-policy RL over real package builds practical.

Example episode. A cross-package fault (symptom in one package, cause in another)
$ chroot /mnt/lfs make -C /sources/kmod-34.2
/usr/bin/ld: undefined reference to `gzread' # symptom: kmod won't link
$ ls -l /usr/lib/libz.so
libz.so -> libbz2.so.1.0 # cause: zlib's dev symlink is mispointed
$ ln -sf libz.so.1.3.1 /usr/lib/libz.so && make && make install
verifier: libkmod compiles, links, and runs (reward 1.0)

2Verifiable reward

Scoring is performed out-of-band by a functional verifier. Inside the repaired system, a small canary program is compiled, executed, and its output compared against ground truth, for example a compress/decompress round-trip that must return the original bytes, an XML parse, or a dynamic-linker resolution. The reward reflects whether the repair actually works, not whether the expected files are present.

The design is deliberately robust to reward hacking. A library stub that only reports the correct version string, while omitting real functionality, does not pass; we implemented that attack ourselves and confirmed every verifier rejects it. The environment exposes two reward signals. A dense shaped reward over intermediate milestones gives even a weak base policy gradient, and a binary reward is used for evaluation.

Verification scales with the fault set rather than gating it. Every package's verifier instantiates the same pattern (compile a canary, perform a real operation, compare the output against ground truth), so a new package requires only a small canary program; the fault classes, milestone reward, and hardening apply unchanged. An automated solvability check then proves each new fault both solvable (the reference fix passes the verifier) and biting (a naive rebuild does not) before it enters the set, so the set grows without accumulating broken or trivial tasks.

3Training result

The environment's own pipeline is sufficient to train a small model from zero capability to reliable execution of the trained repairs. We apply a supervised warm-start on 111 verifier-passed reference trajectories covering every fault type, then run 9 rounds of GRPO from that checkpoint on 8 seeds drawn from three single-fault classes. Total compute is a single GPU at $0.16/hour for a few hours. We state precisely what the evaluation measures. Instances are deterministic and the warm-start demonstrates each fault type, so Table 1 reports training-distribution mastery. The warm-start teaches the repairs but executes them unreliably (3/10 even on demonstrated tasks); GRPO consolidates them into 90% reliable, verifier-passing repairs, executed end-to-end at sampling temperature. Figure 1 shows the learning curve.

The shaped rewards diagnose the failure modes precisely. On the unseen package (shaped 0.06) the policy cannot even execute its procedure. It applies repairs for faults that are not present and uses another package's build recipe. On the trained package with an unseen fault (shaped 0.75, uniformly across episodes) the procedure executes flawlessly. The rebuild succeeds and every intermediate milestone passes, but the fault survives it, and the policy never checks its own repair before declaring completion. Stated plainly, what this recipe produced is a reliable executor of demonstrated repair procedures, not a debugger. It has the procedure but not the loop, with no verification of its own fix and no diagnosis when the fix is not enough. One caveat is that the unseen fault types are hard-tier by construction, so that cell conflates novelty with difficulty; a frontier model solves both in ~9 commands. The environment measures this whole matrix directly, and closing it (multi-package curricula, colder task prompts, larger policies) is the open training question the generator's cheap package scaling exists to serve.

4Difficulty calibration

To place the environment on a capability scale, we evaluated a frontier model (Claude) on progressively harder faults, each further from a self-documenting error message. It solved all of them in 8 to 11 commands, including the three hardest constructions we could design.

  • silent wrong output with no build error;
  • a fault that a rebuild does not clear, because a preloaded shim intercepts the freshly-built library; and
  • a defect planted in the source tarball, with no clean copy available to compare against.

We report this plainly. The current fault set is within the reach of a frontier model, so the environment's value today is in training and evaluating open models, where it produces a capability gradient that Section 3 measures at both ends, 100% zero-shot for the frontier model and 0% for a trained 3B on a never-seen package, with every rung between them (base, warm-start, RL-consolidated) resolved. Whether single-package faults can be made frontier-hard is, on present evidence, an open question rather than a promise.

5Integration and cost

  • FormatPackaged as a verifiers environment (Prime Intellect's format) and validated end-to-end against a live model endpoint, in addition to the native harness.
  • DeterminismSeed-to-instance mapping is fixed; the same seed reproduces the same fault and verifier.
  • FootprintRootless sandbox, ~1.6 s/episode, ~200 MB per instance, no root or VM. A full training run cost about $0.50 of GPU.
  • IncludedThe environment, the fault generator, the training pipeline (warm-start and GRPO), the trained adapters, and the calibration harness and data.
  • ScalingMechanical, verification included. The generator is parameterised over packages and fault types, and each package's verifier is an instance of one canary pattern. The most recent package addition (zstd) consisted of a 56-line verifier and one taxonomy entry, and yielded nine new faults, each automatically proven solvable and biting by CI. Eight further library packages are already staged in the source tree.

6Fit and availability

lfs-rl is intended for teams and platforms that train or evaluate open models on agentic, coding, or operations tasks, and for RL-environment marketplaces that require verifiable, hard-to-game rewards and reproducible, drop-in environments. It satisfies those requirements directly. The reward is functional and resists gaming, the packaging is standard, and the training result is reproducible from the included pipeline.

The full environment, pipeline, adapters, and calibration data are available for evaluation. We welcome a technical review and a conversation about the difficulty and scale targets that would make it useful to you.
